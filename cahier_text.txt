1. Presentation du projet
1.1 Contexte
La gestion manuelle des pharmacies en Afrique de l'Ouest reste la norme dans de nombreux etablissements. Elle engendre des ruptures de stock non detectees a temps, des pertes liees aux medicaments expires, des erreurs de caisse et une absence totale de traçabilite. PharmacyManager BF est une solution fullstack (web + mobile + API) destinee a digitaliser completement les operations d'une officine.
1.2 Objectif general
Concevoir et deployer un systeme d'information complet pour la gestion d'une pharmacie : inventaire des medicaments, suivi des dates d'expiration, enregistrement des ventes, alertes de stock bas, et tableau de bord analytique pour l'administration.
1.3 Perimetre
Application web d'administration (NextJS)
Application mobile pour les pharmaciens et caissiers (Flutter)
API REST backend (NestJS)
Base de donnees (MongoDB)
Deploiement en production

3. Gestion des roles et authentification
3.1 Roles utilisateurs
Le systeme implemente une authentification JWT avec trois niveaux d'acces distincts.
Role
Permissions
Admin
Acces total : gestion utilisateurs, configuration systeme, rapports globaux, suppression
Pharmacien
Gestion inventaire, ventes, alertes stock, consultation rapports
Caissier
Enregistrement ventes uniquement, consultation stock, impression recu
3.2 Authentification
JWT stocke en HttpOnly cookie (web) et SecureStorage (mobile)
Refresh token avec expiration 7 jours, access token 15 minutes
Bcrypt pour le hashage des mots de passe (salt rounds: 12)
Middleware de protection de routes par role sur NestJS (Guards)
Deconnexion forcee en cas de token invalide ou expire

6. API REST Backend (NestJS)
6.1 Architecture
Architecture modulaire NestJS : AuthModule, MedicineModule, SaleModule, StockModule, UserModule, AlertModule
Documentation Swagger auto-generee accessible via /api/docs
Validation des donnees avec class-validator et class-transformer
Gestion des erreurs centralisee via ExceptionFilter global
Logging avec Winston : chaque requete loggee avec methode, path, status, duree

6.3 Securite API
JWT Guard sur toutes les routes protegees
Role Guard avec decorateur @Roles() sur les endpoints sensibles
Rate limiting : 100 requetes/minute par IP (nestjs-throttler)
CORS configure pour les domaines frontend uniquement
Helmet.js pour les headers de securite HTTP
Validation stricte des DTOs - aucun champ non declare accepte

8. Securite et bonnes pratiques
Variables d'environnement via .env - aucune credential en dur dans le code
JWT_SECRET de minimum 256 bits genere aleatoirement
HTTPS obligatoire en production (certificat Let's Encrypt via Render/Railway)
Soft delete uniquement - aucune suppression definitive de donnees critiques
Toutes les entrees utilisateurs validees cote serveur (class-validator)
Logs d'audit pour toute modification de stock ou annulation de vente
Backup automatique MongoDB via Atlas ou script cron quotidien
